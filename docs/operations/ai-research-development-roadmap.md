## 0. 本書の目的

本書は、Claude Fable 5、ChatGPT、Codex、GitHub、Linearを用いて、研究上の妥当性とソフトウェア実装の整合性を維持しながら、個別ルート提示学習モデルの設計、実装、評価、引継ぎを進めるための手順書である。

本書が達成すべき状態は次のとおりである。

- 研究上の重要判断がAIの会話内だけに残らない
- 確定事項、暫定仕様、未確定事項、実装状態が混同されない
- 研究目的から仕様、コード、テスト、ログ、評価指標まで追跡できる
- AIが実装上の都合で研究仕様を確定しない
- MVP範囲を不用意に拡張しない
- 個別化ロジック、説明可能性、評価可能性、再現性を優先する
- 実装結果や小規模評価を過大に主張しない
- Fable 5の利用終了後もChatGPT、Codex、GitHub、Linearで継続できる
- 第三者が研究判断、実装状態、残課題を再現できる
- 運用作業が研究そのものを圧迫しない

## 1. 基本原則

### 1.1 判断主体と正本

ChatGPTやClaudeは研究判断を支援するが、研究仕様を最終決定しない。

| 対象 | 判断・更新主体 | 正本 |
|---|---|---|
| 研究目的・研究質問 | 研究者本人、必要に応じて指導教員 | GitHub `docs/research/` |
| 確定事項・判断理由 | 研究者本人 | `01-confirmed-decisions.md`、`09-decision-log.md` |
| 未確定事項 | 研究者本人 | `02-open-questions.md` |
| MVP仕様・学習モデル | 研究者本人 | `03-mvp-scope.md`、`04-learning-model.md` |
| 評価計画 | 研究者本人、指導教員 | `05-evaluation-plan.md` |
| コード・テスト・CI | 実装担当、研究者本人が受入確認 | GitHub |
| 実装状態 | 実装担当案、研究者本人が承認 | `06-implementation-status.md` |
| 監査結果 | 監査担当 | `docs/audit/` |
| 未完了の実行タスク | 研究者本人 | Linear |
| AI会話 | 補助情報 | 正本にしない |

### 1.2 情報の優先順位

矛盾がある場合、次の順で扱う。

1. 現在の会話で研究者本人が明示した最新判断
2. `01-confirmed-decisions.md`
3. `09-decision-log.md`の最新有効判断
4. `03-mvp-scope.md`
5. 履修計画書・指導教員との合意
6. `00-research-overview.md`
7. その他の最新研究資料
8. 現在のコード
9. 古い文書・古い会話

コードは現在の実装状態を示す一次情報であり、研究仕様の正解とは限らない。

### 1.3 状態ラベル

文書では、少なくとも次を区別する。

- `[確認済み事実]`
- `[コード存在確認済み]`
- `[確定事項]`
- `[暫定仕様]`
- `[提案]`
- `[推測]`
- `[未確定]`
- `[部分実装]`
- `[未実装]`
- `[撤回済み]`
- `[確認不能]`

### 1.4 禁止事項

- AIの会話だけで重要判断を完結させる
- 固定値が動作したことを研究仕様の妥当性とみなす
- 未確定事項を実装者が既定値で補完する
- 比較実験なしに固定ルートより有効と断定する
- 小規模主観評価から長期的学習効果を断定する
- 研究目的に寄与しない技術移行やUI全面改修を優先する
- 実行していないテストや検証を成功と報告する
- 個人情報を含む生データをAIへ入力する
- 評価開始後に中核仕様を無記録で変更する

## 2. ツールごとの責任境界

### 2.1 研究者本人

- 研究方針、仕様、採否、優先順位を決定する
- 必要な事項を指導教員へ確認する
- AI成果物を承認または却下する
- GitHub正本とLinearを最新に保つ
- 評価開始可否を判断する

### 2.2 指導教員

原則として次を確認する。

- 研究質問
- 評価対象者、人数、募集方法
- 比較条件
- アンケート項目
- 研究上の主張範囲
- 同意、匿名化、保存、削除などの倫理・データ管理
- 研究計画からの重要な変更

### 2.3 ChatGPT

- 研究資料・GitHub・評価計画の横断整理
- 矛盾検出
- 選択肢と影響の比較
- decision-log追記案
- Linear Issueへの分解
- Claude、Codex成果物の再監査
- 研究上の主張範囲の検査

ChatGPTは最終承認者ではない。

### 2.4 Claude Fable 5

- 高難度の横断監査
- 中核仕様案の作成
- 複数文書・コード間のトレーサビリティ確認
- 中核実装と独立レビュー
- 引継ぎ成果物の整備

未確定事項を勝手に確定してはならない。

### 2.5 Codex

- 確定済み仕様に基づく実装
- 型定義、純粋関数、テスト
- ID参照整合性検証
- PR作成
- 確定済み形式に基づく文書更新

Issue内に研究者判断が残っている場合、実装を開始せず差し戻す。

### 2.6 GitHub

- 研究文書、コード、テスト、履歴、PR、CIの正本
- 判断主体ではなく、記録・版管理基盤

### 2.7 Linear

- 実行可能な未完了作業の正本
- 研究仕様本文を複製しない
- GitHub文書、要件ID、PR、コミットを参照する

# 5. 実行ロードマップ

## Phase 0：バックアップと基準点

### 開始条件

- リポジトリへアクセスできる
- 現在の作業状態を確認できる

### 作業

1. 最新`main`を取得する
2. 現在ブランチ、HEAD SHA、`git status`を記録する
3. ロックファイルとパッケージマネージャを特定する
4. 依存関係を導入する
5. 利用可能なbuild/test/typecheck/lintコマンドを確認する
6. 現状で実行できる検証を実施する
7. 作業ブランチを作成する
8. 秘密情報や未追跡の個人データがないことを確認する

### 成果物

`docs/audit/baseline-YYYYMMDD.md`または専用Linear Issue

記録項目：

- 日時
- 実行者
- ブランチ
- HEAD SHA
- `git status`
- ロックファイル
- パッケージマネージャと版
- 実行コマンド
- 成功・失敗
- 既知問題
- 作業ブランチ
- 秘密情報確認

### 完了条件

- 第三者が基準コミットと実行コマンドを特定できる
- 作業ブランチが存在する
- 現状の失敗が後続変更によるものと混同されない

### 承認者

研究者本人

### 差し戻し条件

- HEAD SHAが記録されていない
- ビルド結果が未記録
- mainへ直接変更している

## Phase 2：トレーサビリティ監査

### 開始条件

- Phase 1完了
- 最新`main`と監査対象SHAが特定されている

### 作業

研究目的、研究質問、MVP仕様、コード、テスト、評価計画を横断監査する。

### 必須成果物

- `context-consistency-audit.md`
- `implementation-gap-analysis.md`
- `requirements-traceability-matrix.md`
- `risk-register.md`
- `recommended-work-order.md`

### 要件ID候補

- `RQ-xxx`：研究質問
- `LM-xxx`：学習モデル
- `MVP-xxx`：MVP要件
- `ERR-xxx`：エラー分類
- `EVAL-xxx`：評価要件
- `DATA-xxx`：研究データ
- `OPS-xxx`：運用要件

正式採用時は既存文書への影響を確認する。

### 完了条件

- 各指摘に根拠ファイル、重大度、影響、修正案がある
- 実装済み、部分実装、モック、未実装が区別される
- 研究目的から評価指標までの主要経路が追跡できる
- 確認不能事項が明記される
- 別セッションまたは別モデルでCritical／Highを再確認する

### 承認者

研究者本人

### 差し戻し条件

- 根拠のない一般論が中心
- コード状態を研究仕様とみなしている
- 監査対象SHAが不明
- 未確認の検証を成功扱いしている

### 開始条件

3. 採用指摘の反映先を決める
4. 正本文書を更新する
5. 実行作業をLinear Issue化する
6. トレーサビリティ行列を更新する
7. 反映コミットSHAを記録する

### 必須表

| 指摘ID | 採否 | 理由 | 承認者 | 反映先 | Linear Issue | コミット |
|---|---|---|---|---|---|---|

### 反映先

- 確定判断：`01-confirmed-decisions.md`

- 実装状態：`06-implementation-status.md`
- 判断経緯：`09-decision-log.md`
- リスク：`risk-register.md`
- 実行作業：Linear

### 完了条件

- 採用された全指摘に反映先またはIssueがある
- 未確定事項が実装Issueへ直接流れていない
- 古い状態記述が残っていない
- Phase 3で扱う論点が限定されている

### 承認者

研究者本人。研究質問、評価、倫理事項は必要に応じて指導教員。

## Phase 4：最小実装・検証基盤

### 開始条件

- Phase 3承認ゲート通過
- 実装Issueが確定済み

### 優先順位

1. 中核ロジックの単体テスト
2. build
3. 必要な型検査
4. CI
5. Lintは変更対象中心

リポジトリ全体の既存Lint修正は、研究目的に必要な場合を除き対象外とする。

### 作業

- テストランナー導入
- `test`スクリプト
- 必要に応じた`typecheck`
- build確認
- 中核テストを実行するCI
- ID参照、循環、到達不能の検査
- 開発手順の文書化

### 完了条件

- 同一コミットでbuildと中核テストを再実行できる
- 失敗時に終了コードが非0
- CIまたは明示的なローカル検証手順がある
- 基盤整備が中核実装以上に肥大化していない

### ロールバック

- 原則としてPR単位でrevert可能にする
- DB変更が必要な場合はmigrationと戻し方を記録する
- 評価データの互換性を壊す変更は別Issueとする

## Phase 5：ルート生成ロジック実装

### 開始条件

- Phase 4完了
- ルート生成仕様が確定済み
- Issueに未確定事項がない

### 実装方針

- MVP 12ノード限定
- UIから独立した純粋関数
- 同じ入力には同じ出力
- 正規IDのみ使用
- 推薦理由を構造化
- 研究仕様にないAI推論を実行時に必須化しない

### 必須テスト

- 診断差による開始ノード差
- 前提未完了ノードを直接推薦しない
- 完了済みノード除外
- テスト未達による復習
- エラーによる復習
- 振り返り反映
- 同点処理
- 入力欠損
- 存在しないID
- 循環
- 到達不能
- 決定性
- 全出力IDがMVP集合内

### 完了条件

- 全受入条件とテストが対応する
- 理由情報が仕様に従う
- モックや未対応ケースが明記される
- `06-implementation-status.md`と行列が更新される

## Phase 6：UI・最小評価ログ接続

### 開始条件

- Phase 5完了
- UIへ渡す契約が確定済み
- 評価に必要な最小ログが定義済み

### 作業

代替手段：

- Supabase
- JSONまたはCSV出力
- 外部フォーム
- 研究者向け簡易エクスポート

### 完了条件

- 推薦内容と理由がUIに表示される
- 入力状態変更で再計算される
- 評価指標ごとに取得元がある
- 代表ユーザーのログを出力できる
- 欠損・重複・研究IDを確認できる

## Phase 7：独立受入監査

### 開始条件

- Phase 6完了
- 受入条件と使用コミットSHAが明確

### 独立性レベル

- Level 1：同一セッション別担当
- Level 2：新規セッション
- Level 3：別モデル
- Level 4：人間確認

Critical／Highの検証は最低Level 2、研究判断はLevel 4とする。

### 監査観点

- 仕様準拠
- 要件IDとの対応
- テスト網羅
- ID整合性
- 循環・到達不能
- 理由情報
- エラー処理
- UI接続
- 評価ログ
- 過大な機能追加
- 個人データ混入
- 文書更新

### 完了条件

- Critical／Highが0、または明示的に受容・Issue化されている
- 代表シナリオだけでなく境界・異常系が確認されている
- `06-implementation-status.md`とトレーサビリティ行列が最新
- 監査対象コミットSHAが記録されている

## Phase 7.5：評価実験開始ゲート

### 判定

- `GO`
- `CONDITIONAL GO`

## Phase 9：評価、分析、論文・発表反映

### 開始条件

- Phase 7.5が`GO`または承認済み`CONDITIONAL GO`

### 作業

- 評価実施
- 逸脱、欠損、途中中止の記録
- 事前に定めた分析
- 自由記述の匿名化
- 結果と解釈の分離
- 限界の整理
- 改善点の整理
- 論文・発表への反映

### 主張上の制約

- 単群評価では相対的優位を主張しない
- 小規模主観評価では長期的離脱率低下を断定しない
- 実装確認と学習効果を区別する
- 統計的検定を行わない場合は探索的結果として扱う
- 仕様変更があった場合は条件を分離する

### 完了条件

- 研究質問ごとに対応する結果がある
- 解釈に使ったデータと仕様版が追跡できる
- 限界と一般化可能範囲が明記される
- 改善案が次Issueまたは将来課題へ整理される

# 10. 定常運用サイクル

```text
相談・不具合・判断発生
  ↓
ChatGPTが研究判断と実装判断を分離
  ↓
未確定ならOQまたは判断Issue
確定済みならLinear実装Issue
  ↓
Codexがブランチで実装・テスト・PR
  ↓
仕様・要件ID・受入条件でレビュー
  ↓
マージ
  ↓
必要箇所だけ文書・行列・実装状態を更新
  ↓
Issue完了承認
```

### 判断発生時

- 選択肢と影響を整理
- 本人が承認
- 必要に応じて指導教員確認
- decision-log更新
- confirmed-decisionsまたはopen-questions更新

### PR反映時

- 受入条件確認
- テスト結果確認
- 仕様逸脱確認
- implementation-status更新
- 行列更新
- LinearへPR・コミット記録

### 長期中断後

1. 最新`main`と未マージPRを確認
2. `00`、`01`、`02`、`06`、`09`、`10`を読む
3. Linear未完了Issueを確認
4. 前回SHAとの差分を確認
5. build/testを再実行
6. 未確定事項の混入を確認
7. 次Issueを1件だけ選ぶ

### 監査トリガー

- 中核仕様確定前
- `routeGenerator`実装完了時
- 評価実験開始前
- 評価計画変更時
- 論文・発表前
- 1か月以上中断後
- Critical／High不具合発生時
- AIまたは技術基盤の大幅変更時

# 11. 運用負荷を抑える原則

必須：

- 確定・未確定・実装状態の分離
- decision-log
- 中核仕様とコード・テストの対応
- 評価ログとの対応
- 研究データ管理
- 評価開始ゲート
- 過大主張防止

縮退可能：

- Lintは変更対象のみ
- CIは中核テストとbuildを優先
- architecture文書は必要に応じて統合
- 独立レビューは中核実装・評価開始前に限定
- トレーサビリティは中核要件のみ
- 軽微な変更で全AI・全承認を要求しない
- 監査文書は毎回全面再生成せず差分更新する
