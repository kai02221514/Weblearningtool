## 目的

診断、進捗、確認テスト、実践課題のエラー、振り返りを正規ノードIDへ統合し、前提関係を満たす順序付き学習ルートと、その選出理由を生成する。

## ルート生成の入力

最低限、次の情報を一つの入力スナップショットとして受け取る。

| 区分 | 主な項目 | 用途 |
| --- | --- | --- |
| カタログ | `catalogVersion`, ノード一覧, 前提関係 | ID検証と前提ノード補完 |
| 診断 | 回答ID、回答値、レベルスコア、判定レベル | 初回の開始候補と基礎復習範囲の決定 |
| 進捗 | `completedNodeIds`, `inProgressNodeId`, 完了日時 | 完了済みの除外、再開位置の決定 |
| テスト | 問題ID、対応ノードID、正誤、得点、試行日時 | 未習得・再確認ノードの抽出 |
| エラー | 正規エラーID、回数、発生日時、対象ノード、解消状況 | つまずきに対する補習ノードの抽出 |
| 振り返り | つまずいたノードID、自己評価、自由記述、日時 | 学習者が自覚した弱点の反映 |
| 制約 | 最大ノード数、利用可能時間、対象領域 | MVPや学習セッションに合わせた切り詰め |

入力には `routeVersion` を指定するか、実行したアルゴリズム側で付与する。研究用途では、ルート生成に使った入力を後から再現できる識別子を保存する。

## ルート生成の出力

出力は単一の次ノードだけでなく、順序付きのルートと説明情報を持つ。

```ts
interface GeneratedRoute {
  routeId: string
  routeVersion: string
  catalogVersion: string
  generatedAt: string
  nextNodeId: string | null
  nodes: Array<{
    nodeId: string
    order: number
    reasonCodes: string[]
    sourceSignals: string[]
    prerequisiteFor?: string[]
  }>
  status: 'active' | 'completed' | 'insufficient-input'
  warnings: string[]
}
```

`reasonCodes` は、少なくとも `DIAGNOSTIC_GAP`、`IN_PROGRESS`、`TEST_INCORRECT`、`ERROR_REMEDIATION`、`REFLECTION_GAP`、`PREREQUISITE`、`REVIEW` を区別する。自由文だけを保存せず、集計可能な理由コードと画面表示用の説明を分離する。

## 生成手順の案

1. 入力のIDを正規化し、未知ID、カタログ版不一致、欠損値を検出する。
2. 進行中ノード、診断上の弱点、テスト誤答、未解消エラー、振り返り上のつまずきから候補ノードを作る。
3. 完了済みノードを候補から除外する。ただし、再発エラーや連続誤答など、復習条件を満たす場合は `REVIEW` として戻す。
4. 各候補の未完了前提ノードを再帰的に追加する。
5. 前提関係を壊さないようトポロジカルソートする。
6. 同一段階の候補は、進行中、未達前提、未解消エラー、テスト誤答、振り返り、診断の順を基本として優先する。
7. 同順位ではエラーの新しさ・反復回数、テスト試行回数、ノードIDの順で決定し、同じ入力から同じ結果を返す。
8. セッション制約に合わせてルートを切り詰め、各ノードの理由を出力する。

重みや優先順は研究上の比較要因になり得るため、実装定数として埋め込む前に確定し、`routeVersion` ごとに追跡する。

## 診断・進捗・テスト・エラー・振り返りの使い方

### 診断

- 初回ルートの開始位置と、基礎ノードを省略できる可能性を決める。
- 総合レベルだけでなく、回答項目をノードへ対応付けて領域別の弱点を扱う。
- 自己申告のみでノードを修了扱いにはせず、必要なら短いプレテストで確認する。

### 進捗

- 進行中ノードを原則として最初の候補にする。
- 修了済みノードを通常ルートから除外する。
- 修了条件、修了日時、修了根拠を保存し、単なる画面閲覧と習得を区別する。

### テスト

- 各問題を一つ以上の正規ノードIDへ紐付ける。
- 誤答ノードとその前提ノードを再学習候補にする。
- 単発誤答と反復誤答を区別し、試行履歴に応じて優先度を調整する。

### エラー

- 実践課題の検出結果を正規エラーIDへ変換し、エラーマッピングの `nodeRefs` から補習候補を得る。
- `nodeRefs.priority`、発生回数、最終発生日時、解消状況を推薦に使う。
- 同一エラーの再発は、関連ノードを修了済みでも復習候補へ戻す根拠にできる。

### 振り返り

- 現在の概念名文字列ではなく、正規ノードID付きの選択肢として保存する。
- 自己申告のつまずきを候補へ加えるが、テスト・エラーと重なった場合に優先度を上げる。
- 自由記述はMVPでは自動判定に直接使わず、評価・質的分析用に扱う。

## フォールバック

- 入力が不足している場合は、未完了の入口ノードから前提順に提示し、`insufficient-input` を返す。
- 全MVPノードが完了し、復習条件もない場合は `nextNodeId: null` と `completed` を返す。
- 未知エラーIDは履歴には残すが、推薦には使わず警告へ含める。
- 前提グラフに循環や参照切れがある場合は、推測でルートを返さず生成エラーとして扱う。

## まだ実装していない点

- 上記の統一入出力型と独立したルート生成関数
- 診断回答・テスト問題・振り返り選択肢から正規ノードIDへの対応表
- エラーIDとノードIDの正規化
- 候補スコア、重み、同順位規則、復習条件
- 前提ノードの再帰補完とトポロジカルソート
- ルート理由コードとルート版の保存
- 入力スナップショット、生成結果、画面表示の接続
- 代表シナリオ、境界値、循環、参照切れに対する自動テスト

現行コードには前提を満たす次候補の抽出、ダッシュボード上の推奨表示、エラー別推薦の断片があるが、固定値と不一致IDを含むため、本仕様のルート生成実装とはみなさない。
