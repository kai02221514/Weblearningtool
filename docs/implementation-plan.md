## Phase 2: ルート生成ロジックの実装

### 作業内容

- [ルート生成仕様](./route-generation-spec.md)をMVP 12ノード、8エラー、`quiz-{nodeId}`、`practice-profile-card` の確定契約へ更新する。
- `routeGenerator` から既存のMVPノード単一定義を参照し、画面・検証・ルート生成で共有する。
- [ルート生成仕様](./route-generation-spec.md)に基づく入出力型を定義する。
- 確認テストをノード単位のデータへ分離し、問題ID、テストID、対象ノードIDを保持する。
- 診断結果から開始候補を選び、進捗済みノードを除外または復習候補として扱う。
- テスト誤答、エラー履歴、振り返りをノード別の再学習優先度へ変換する。
- 候補ノードの前提ノードを再帰的に補完し、トポロジカル順で並べる。
- 同じ入力に対して同じルートを返す決定的な優先順位と同順位規則を定義する。
- ルートの各要素に、選出理由と根拠となったシグナルを付ける。
- ダッシュボードと学習開始処理を生成ルートへ接続する。

### 完了条件

- 診断のみ、進捗あり、テスト誤答あり、反復エラーあり、振り返りありの代表ケースで期待ルートを生成できる。
- 出力ルートに未達の前提ノード、重複ノード、未知IDが含まれない。
- 生成結果が決定的で、各ノードの推薦理由を表示または記録できる。
- 入力不備、空ルート、全ノード完了時のフォールバックが定義・テストされている。
- 画面上の「次に学ぶ」表示と実際の学習開始ノードが一致する。

### 次に作成するIssue

**件名:** `route-generation-spec.mdをMVP 12ノード契約へ更新し、決定的なrouteGeneratorを実装する`

**受け入れ条件:**

- MVP集合外のノードとMVP外エラーを入力しても推薦ルートへ混入しない。
- `completedNodeIds` と `inProgressNodeId` が正規ノードIDとして検証される。
- `quiz-{nodeId}` の誤答と `practice-profile-card` のMVPエラーから対象ノードを抽出できる。
- 未完了前提を再帰補完し、前提順を守る。
- 同じ入力から同じ順序、理由コード、警告を返す。
- 未知ID、参照切れ、循環、全完了、空入力のテストがある。

## Phase 3: 進捗・診断・エラー履歴の保存

### 作業内容

- ユーザー、診断回答、ノード進捗、テスト試行、エラーイベント、振り返り、生成ルートの保存モデルを定義する。
- タイムスタンプ、`catalogVersion`、`routeVersion`、アプリ版を保存する。
- 保存APIと読込APIを実装し、サインイン後に学習状態を復元する。
- 同一イベントの重複保存を防ぐため、イベントIDまたは冪等キーを導入する。
- エラー履歴に回数と最終発生時刻だけでなく、対象ノード、検出方法、解消状況を保持する。
- 研究用途の出力形式と、削除・匿名化の手順を定義する。

### 完了条件

- 再読み込み・再ログイン後も診断、進捗、最新ルート、履歴を復元できる。
- テスト再受験や同一エラーの複数発生を時系列で区別できる。
- ルート生成時に利用した入力スナップショットと生成結果を対応付けられる。
- 保存失敗時に学習操作を失わない再試行または明示的エラー処理がある。
- 個人情報を含む項目、保持期間、削除方法が評価計画と一致している。

## Phase 4: 評価アンケートとデモ導線整備

### 作業内容

- [評価計画](./evaluation-plan.md)に基づく事前・事後アンケートを実装する。
- 固定ルートと生成ルートを比較する場合の条件割付と記録方法を実装する。
- 初回診断から学習、テスト、実践、振り返り、事後評価まで中断なく進めるデモ導線を整備する。
- デモ用の初期データ、想定シナリオ、リセット手順を用意する。
- 回答必須・任意、同意取得、データ利用目的、撤回方法を画面に反映する。
- 研究データの欠損確認と匿名化済み出力を行えるようにする。

### 完了条件

- 新規参加者が説明だけで一連の評価フローを完了できる。
- 主要評価項目と行動ログを参加者・セッション・ルート版に紐付けられる。
- 比較条件を採用する場合、割付結果と提示ルートを再現できる。
- デモのリセットと再実行ができ、前回データが混入しない。
- 同意、匿名化、保持期間、データ出力について研究上の確認が完了している。

## Phase間の依存関係

Phase 1のIDとデータ範囲をPhase 2以降の契約とする。Phase 2とPhase 3は入出力仕様確定後に一部並行できるが、評価用データ項目を確定せずに保存モデルを固定しない。Phase 4開始前に、評価指標へ必要なイベントがPhase 3で取得できることを確認する。
