## 1. 文書間の矛盾(C系)

### C-1 文書二系統が併存し、どちらが正か宣言されていない【Critical】

docs/直下(mvp-scope.md, evaluation-plan.md, implementation-plan.md, route-generation-spec.md, research-status.md/2026-06-19)と docs/research/00〜10(2026-07-02監査スナップショット)の二系統がある。優先順位の手掛かりが相互に矛盾する。

- research/00「情報の優先順位」は research系(01→09→03)を上位に置き、docs/直下文書をリストに含めない。
- research/10-handover は「docs/mvp-scope.md: 現在有効なMVP仕様」と旧系統を正と明言。09-decision-log の D-001〜D-005 の根拠もすべて旧系統。
- research-status.md「同期ルール」は evaluation-plan.md / route-generation-spec.md を更新先(=正)とする運用を規定。

MVPスコープは両系統で内容一致のため実害は小さいが、評価計画(C-2)とルート生成契約(C-4)は内容が乖離しており、正の宣言がないままでは後続実装が誤った契約を採用しうる。

**推奨**: 各文書の冒頭に「正/参考(superseded)」ステータスを明記し、decision-logに宣言を1件追加する。

### C-2 評価計画の内容乖離(旧 evaluation-plan.md vs research/05・08)【High】

- 旧 docs/evaluation-plan.md: 「理解度向上」「エラー再発・修正時間の減少」を確認する効果測定志向。プレ・ポストテスト、遅延テスト、自己効力感尺度、群間比較・ランダム化まで想定。
- 新 research/05: 納得感・受容性・有用性の主観評価中心。research/08「主張範囲」[確定事項]は「主観評価のみの場合、学習成果向上ではなく受容性・納得感・継続支援可能性として述べる」と旧文書の目的設定を実質否定。
- 一方、旧文書の「保存すべきデータ」(同意状態、仮名参加者ID、入力スナップショットID、catalogVersion/routeVersion、ルート差分等)は詳細かつ有用だが、新系統に引き継がれておらず、評価ログの確定仕様がどこにも存在しない(G-2)。
- 旧文書は自ら「文書ステータス: 初期案」と申告している。

**推奨**: research/05を正とし、旧evaluation-plan.mdの「保存すべきデータ」節のみOQ-009の検討材料として明示的に引き継ぐ。

### C-3 docs/mvp-scope.md と research/03 の役割分担が暗黙【Medium】

12ノード・前提・8エラー対応・ID規則・色指定は完全一致(矛盾なし)。ただし quiz ID一覧、除外14ノード表、MVP外6エラー一覧、practice-profile-card の対象10ノードは docs/mvp-scope.md のみに存在し、research系だけを読むと見えない。

### C-4 route-generation-spec.md の確定度矛盾【High】

旧specは GeneratedRoute 型、reasonCodes 最低集合(DIAGNOSTIC_GAP等7種)、優先順位案「進行中>未達前提>未解消エラー>テスト誤答>振り返り>診断」、同順位規則、フォールバックまで具体定義する。research系(OQ-005, 04-learning-model)はこれらを「未確定」「検討用案」と明言し、10-handover は「未確定のreasonCode等をCodexが独自に確定する」ことを禁じる。しかし implementation-plan Phase 2 と mvp-scope.md 次Issue案は旧spec契約への更新を前提としており、**旧spec案が無審査で既定値として滑り込むリスク**がある。

**推奨**: OQ-005決定時に旧spec案の採否を項目別に明示する(全否定する必要はない。たたき台として有効)。

### C-5 「実装済み」ラベルの2水準運用【Medium】

research-status.md は画面コンポーネント・固定問題テスト・簡易エラー検出を「実装済み」に分類。research/01・06の定義(「実装済み=正規データで動作し受入条件の検証が通った状態」)では同じ項目群が「コード存在確認済み」に降格される。同一リポジトリ内で同一機能の状態ラベルが2水準併存。

### C-6 参照切れ: 存在しない監査文書への参照【Medium】

research/00 は `11-independent-audit.md`、`12-independent-reaudit.md`、`14-independent-reaudit.md` を参照するが、docs/research/ には00〜10しか存在しない(GitHub APIで全ファイル確認、truncated:false)。「最新監査結果は14を参照」という指示が実行不能。

### C-7 根拠を追跡できない記述【Medium】

OQ-010 の「履修計画書上の Next.js + Deno/Hono + Supabase 構想」は、履修計画書docx抽出本文に該当記述がない(技術スタックの記載自体がない)。別版の計画書または口頭合意由来の可能性があり、根拠が追跡できない。
