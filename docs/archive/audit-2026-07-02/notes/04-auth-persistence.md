## 1. 認証フロー(サインアップ / ログイン / セッション維持)

### 実態
- **サインアップ**: フロント(`Auth.tsx`)→`utils/auth.ts` の `signup()`→Edge Function `POST /make-server-f3d88633/signup`→`supabase.auth.admin.createUser()`(service role 権限)でユーザー作成。`email_confirm: true` によりメール確認を自動スキップ(メールサーバ未設定のため)。作成後 KV に `user:{id}` を保存。
- **ログイン**: `signin()`→`POST /signin`→`supabase.auth.signInWithPassword()`。成功時に `access_token` / `userId` / `name` を返却。
- **セッション維持**: **存在しない**。フロントに supabase-js クライアントは無く、`accessToken` は `App.tsx` の `useState`(React state)にのみ保持。localStorage / cookie への保存なし、トークンリフレッシュなし、ページリロードで即ログアウト状態に戻る。ログアウト機能も無い。
- **サインアップ直後は accessToken を取得しない**: `handleSignupSuccess(email, name, userId)` はトークンなしで survey フェーズへ遷移。つまり新規ユーザーは認可必須の `/profile` を呼べない状態で学習フローに入る。
- **エラー処理**: フロントは try/catch + Alert 表示(`Auth.tsx` `handleSubmit`)。サーバは 400/401/500 を返し `console.log` に記録。ログイン失敗は「メールアドレスまたはパスワードが正しくありません」に丸めており列挙攻撃対策としては妥当。
- **匿名利用**: 不可。`App.tsx` の初期 `phase` が `'auth'` で、認証を通らないと他フェーズに遷移できない。ゲストモードやスキップ導線は無い。

### 実装状態
**部分実装**(サインアップ/ログインの API 呼び出しは本物の Supabase Auth に到達するが、セッション維持・ログアウト・トークン更新は未実装)

### 根拠
- `src/utils/auth.ts`: `const API_URL = \`https://${projectId}.supabase.co/functions/v1/make-server-f3d88633\`` — 実 URL への fetch。
- `src/supabase/functions/server/index.tsx` signup: `await supabase.auth.admin.createUser({ email, password, user_metadata: { name }, email_confirm: true })`
- 同 signin: `await supabase.auth.signInWithPassword({ email, password })` → `accessToken: data.session.access_token`
- `src/App.tsx`: `const [userData, setUserData] = useState<UserData | null>(null)` — 永続化なし。`localStorage`/`sessionStorage`/`onAuthStateChange` の使用箇所なし。

### 問題
- リロードで全セッション・全進捗が消える(研究ツールとして致命的)。
- サインアップフローでトークン未取得のため、初回ユーザーは認可付き API を一切呼べない。
- ログアウト・パスワードリセット・トークン失効処理なし。
- signin を service role クライアントで実行している(動作はするが、anon クライアントで行うのが通例)。

### 実装状態
**実装済み**(ラッパー自体は完全。ただし利用は 2 キー種のみ)

### 根拠

## 9. RLS / セキュリティ(service role key・クライアント直接アクセス範囲)

### 実態
- **service role key**: Edge Function 内でのみ `Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')` から取得(`index.tsx` と `kv_store.tsx` の 2 箇所)。コードへのハードコードは無く、クライアントバンドルには含まれない。**この点は適切**。ただしサーバは全 DB 操作を service role(RLS バイパス)で行うため、認可はアプリコード(profile のトークン検証)だけが頼り。
- **クライアントの直接アクセス範囲**: フロントは supabase-js を使わず、Edge Function への fetch のみ(anon key は Functions ゲートウェイ通過用)。コード上、クライアントから DB への直接アクセス経路は無い。
- **ただし**: anon key と projectId が公開されているため(項目2)、`kv_store_f3d88633` テーブルに RLS が有効化されていなければ、第三者が PostgREST(`https://<projectId>.supabase.co/rest/v1/kv_store_f3d88633`)経由で**全ユーザーの email・name・profile を読み書き可能**。RLS 設定はコードから確認不能(マイグレーション/SQL ファイルなし)。Figma Make テンプレートのコメントにも RLS への言及がない。→ **要ダッシュボード確認の最重要未検証リスク**。
- その他: CORS `origin: "*"`、signin/signup にレート制限なし(ブルートフォース・アカウント量産可)、パスワード最小長 6 文字(フロントのみで検証、サーバ側は未検証)。

### 実装状態
**部分実装**(鍵の分離は適切、RLS 検証不能、レート制限等なし)

### 根拠
- `index.tsx`: `createClient(Deno.env.get('SUPABASE_URL') ?? '', Deno.env.get('SUPABASE_SERVICE_ROLE_KEY') ?? '')`
- `kv_store.tsx`: 同様に service role で毎回クライアント生成。
- リポジトリに SQL マイグレーション・RLS ポリシー定義ファイルが見当たらない(取得対象ファイルおよび Figma Make 生成物の構成から)。

### 問題
- **[要確認・高リスク]** kv_store テーブルの RLS 無効なら、公開済み anon key で PII 全件が漏洩する。
- パスワードポリシーがサーバ側で未強制(Supabase デフォルト 6 文字頼み)。
- 認可付きエンドポイントが 1 本しかないため、今後進捗 API を足す際に認可パターンの統一が必要。
