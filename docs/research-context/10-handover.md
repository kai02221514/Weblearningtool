# AI引継ぎ手順

## 最初に読む順序

1. `00-research-overview.md`
2. `01-confirmed-decisions.md`
3. `02-open-questions.md`
4. 作業に応じて `03`〜`08`
5. 判断変更時は `09-decision-log.md`

## 作業開始前チェック

- 最新判断と矛盾していないか。
- 確定事項、暫定仕様、未確定を混同していないか。
- コード状態を研究仕様として誤認していないか。
- MVP 12ノードと正規IDを使用しているか。
- 研究目的に寄与しない大規模変更ではないか。
- 受入条件と検証方法が定義されているか。

## Claude / ChatGPTに任せる作業

- 資料横断の矛盾検出
- 研究質問・評価案の比較
- 仕様書・decision-logの更新案
- 実装タスクの目的、対象、対象外、受入条件への分解
- 結果の主張範囲とリスクの検査

## Codexに任せる作業

確定済み仕様に基づく以下の実装。

- 型定義、純粋関数、テスト
- ID参照整合性検証
- routeGenerator
- 理由表示
- 永続化
- 評価ログ・出力

Codexは診断重み、評価条件、研究質問を独自に確定してはいけない。

## 次の推奨Issue

### Issue 1: ルート生成契約の確定と実装
- 目的: MVP 12ノード限定で決定的なルートを生成する。
- 変更対象: domain型、routeGenerator、テスト、仕様文書
- 対象外: UI全面改修、全63ノード対応、AI解析
- 受入条件:
  - 入力が同じなら出力が同じ
  - 前提未完了ノードを直接推薦しない
  - 全出力IDがMVP集合に属する
  - 各推薦にreasonCodeがある
  - 循環・参照切れテストが通る
- 検証: 単体テストと代表シナリオ

### Issue 2: 確認テストのノード対応
- 目的: `quiz-{nodeId}` と問題・結果を正規ノードへ接続する。
- 受入条件: 得点、合否、誤答、対象ノードを保存可能

### Issue 3: 学習履歴の永続化
- 目的: 再読み込み後も診断、進捗、テスト、エラー、振り返り、ルートを復元する。
- 受入条件: 匿名研究IDで一連の履歴を追跡可能

## 判断が発生した場合

1. `02-open-questions.md` を更新する。
2. `01-confirmed-decisions.md` に現在有効な判断を反映する。
3. `09-decision-log.md` に理由・根拠・影響・再検討条件を追記する。
4. 実装影響があれば `06-implementation-status.md` と対象仕様を更新する。

## 禁止事項

- チャットだけで重要判断を完結させる。
- 固定値が動くことを研究仕様の妥当性とみなす。
- 比較実験なしで固定ルートより有効と断定する。
- 小規模主観評価から長期学習効果を断定する。
- 研究目的に寄与しない技術移行や機能追加を優先する。
