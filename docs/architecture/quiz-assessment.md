## 6. 確認テストの目的

確認テストの目的は次の4点に限定する。

1. 学習者が当該ノードの最低限の到達目標を理解しているかを確認する。
2. 不合格時に、当該ノードまたは関連する前提ノードへの復習を促す。
3. `docs/architecture/route-generation.md`の`quizResults[].passed`へ接続できる論理入力を生成する。
4. 研究評価時に、学習過程の補助指標として確認テスト結果を参照できるようにする。

[非目的] 確認テストは、長期保持、実践力、固定ルートに対する優位性、学習者能力の包括的測定を単独で主張するための根拠ではない。

## 10. 問題数

[採用候補] MVPでは、各ノード3問を基本単位とする。

理由:

- 初学者向けMVPとして、1ノードあたりの負担を短く保てる。
- ルート生成に必要な`passed`を得るには最低限の複数証拠が必要であり、1問だけでは偶然正答の影響が大きい。
- 実装と教材整備の負荷をKAI-15で管理できる範囲に抑えられる。

[制約] 3問は信頼性の高い測定を保証しない。研究評価では、確認テストを能力測定の主要指標として過大解釈してはならない。必要に応じて予備試行後に4問または5問へ変更する。

## 11. 問題形式

[採用候補] MVPで許可する問題形式は次の4種類とする。

| 形式 | 用途 | 採点方針 | 採用理由 | 注意 |
| --- | --- | --- | --- | --- |
| 単一選択 | 概念分類、代表例の判別 | 選択肢ID一致 | 初学者負担が低い。 | 偶然正答の可能性がある。 |
| 複数選択 | 複数条件の確認、誤概念の検出 | 正答集合の完全一致 | 一問で複数観点を確認できる。 | 採点表示が分かりにくくなりやすい。 |
| コード並べ替え | HTML構造、CSS記述順序の確認 | ID列一致 | 構文理解を確認しやすい。 | 操作UIが必要で実装負荷が上がる。 |
| 短いコード補完 | タグ、属性、CSS宣言の最小記述 | 正規化後の許容解一致 | 記述力の最小確認に使える。 | 表記揺れと全角半角の扱いが必要である。 |

[対象外] 自由記述の自然言語採点、長いコード入力、自動実行によるDOM/CSS判定はMVP確認テストの対象外とする。実践課題の自動判定範囲はOQ-007で扱う。

## 12. 採点規則

[採用候補] 各問題は1点、各ノードの満点は3点とする。

| 形式 | 正答条件 | 部分点 |
| --- | --- | --- |
| 単一選択 | 選択肢IDが正答IDと一致する。 | なし |
| 複数選択 | 選択肢ID集合が正答ID集合と完全一致する。 | MVPではなし |
| コード並べ替え | 提出された要素ID列が正答ID列と一致する。 | MVPではなし |
| 短いコード補完 | 空白、改行、大文字小文字など仕様で定めた正規化後、許容解集合のいずれかと一致する。 | MVPではなし |

[注意] 部分点を入れると評価解釈とルート接続が複雑になるため、MVPでは導入しない。部分点は予備試行で誤答が多い場合の再検討候補とする。

## 13. 合格判定

[採用候補] 各ノード3問中2問以上正答で合格とする。

```text
passed = score >= 2
maxScore = 3
passScore = 2
```

この規則では、割合表示は補助情報であり、合格判定の主条件ではない。3問構成では2問正答が66.7%相当になるため、`percentage >= 70`のような割合基準を主条件にすると、2問正答が不合格になる。現行実装はこの状態であり、研究仕様としては未確定である。

[設計判断] 2/3合格は、形成的評価として復習を促しながら進行不能を避けるための採用候補である。十分な測定信頼性を示すものではない。

[変更条件] 予備試行で次のいずれかが確認された場合、合格点または問題数を再検討する。

- 多くの学習者が偶然正答により通過している。
- 1問の誤答が本質的な未理解を示す設計になっている。
- 2/3合格後に実践課題で同じ概念のエラーが多発する。
- 3問では到達目標を十分にカバーできない。

## 14. 再受験規則

[採用候補] 不合格後の再受験は許可する。回数制限は設けない。

条件:

- 不合格時は、解説と復習推薦を見た後で再受験できる。
- 各受験には`attemptNumber`を付与する。
- 受験履歴は論理上すべて保持する。ただし保存先、保持期間、削除手順はKAI-12で決める。
- 再受験時に問題を完全に同一にするか、同一到達目標内の別問題にするかはKAI-15で決める。

[理由] 回数制限は初学者の学習機会を減らす可能性がある。確認テストを形成的評価として扱う場合、不合格後に復習して再挑戦できることを優先する。

[リスク] 無制限再受験は丸暗記による通過を生む可能性がある。このリスクは、受験回数、所要時間、誤答推移を評価時に別指標として見ることで緩和する。

## 15. 合格後再受験の扱い

[採用候補] 一度合格した同一`questionSetVersion`の確認テストは、後続の任意再受験で不合格になっても進行状態を自動で未合格へ戻さない。

| 状態 | 進行上の扱い | 評価ログ上の扱い |
| --- | --- | --- |
| 初回不合格 | `passed=false` | 不合格受験として記録候補 |
| 不合格後に合格 | `passed=true` | 不合格から合格への遷移として記録候補 |
| 合格後に任意再受験して不合格 | `passed=true`を維持 | 最新受験不合格として別途記録候補 |
| 問題集合版が変わった後 | 研究者判断まで旧合格を履歴として保持 | 再確認フラグ導入の要否をKAI-12/KAI-15で判断 |

[設計判断] 合格後の任意再受験失敗で進行状態を取り消すと、学習者の混乱と状態管理の複雑化が大きい。一方で評価分析では最新受験と初回合格を分ける必要がある。

## 16. 不合格時のルート接続

[採用候補] 不合格時の主推薦は、対象ノード自身とする。

```text
failed quiz-html-021 -> mainReviewNodeId = html-021
```

`docs/architecture/route-generation.md`では、確認テスト不合格は`QUIZ_FAILED`として、対象ノードを復習候補にできる。KAI-11では、その入力となる`passed=false`と`nodeId`を定義する。

接続規則:

- 未受験は不合格ではない。未受験は`quizResults`に存在しない、または`QuizState`が未作成の状態として扱う。
- 不合格は`passed=false`としてルート生成入力へ渡せる。
- 主推薦は対象ノード自身とする。
- 補助復習候補は、誤答した問題に紐づく前提ノードから生成する。
- 前提未完了ノードを飛ばす推薦をしない制約は、ルート生成側の前提補完規則と合わせて満たす。

## 17. 誤答と復習ノードの対応

[採用候補] 各問題は、主対象ノードと補助前提ノード候補を持つ。

```ts
interface QuizQuestionSpec {
  questionId: string
  nodeId: string
  relatedPrerequisiteNodeIds: string[]
  type: 'singleChoice' | 'multipleChoice' | 'codeOrdering' | 'shortCodeFill'
  questionSetVersion: string
}
```

誤答時の推薦候補生成:

1. 誤答した問題の`nodeId`を主復習候補にする。
2. 誤答した問題の`relatedPrerequisiteNodeIds`を補助復習候補にする。
3. 重複を除外する。
4. MVP 12ノード外のIDは無効として扱う。
5. 前提未完了ノードの扱いは`route-generation.md`の前提補完規則に従う。

[例] `html-022`の属性問題で、属性の引用符だけでなく要素構造の理解不足が疑われる場合、主推薦は`html-022`、補助候補は`html-020`とする。

## 18. 解説・フィードバック規則

[採用候補] 各問題には、正答時と誤答時の短い解説を用意する。

最低要件:

- 結果画面で、正誤、正答、学習者の回答、解説を表示する。
- 誤答時は、どの概念を復習すべきかを示す。
- 不合格時は、再受験前に解説または復習導線を見られるようにする。
- 解説文はKAI-15で作成し、問題集合版と同じ版管理対象にする。

[注意] フィードバックは、内容と提示方法によって効果が変わる。単に正答を表示するだけでは、学習改善につながるとは限らない。

## 19. 問題集合の版管理

[採用候補] すべての問題集合に`questionSetVersion`を付与する。

```text
questionSetVersion = quiz-{nodeId}/v{major}.{minor}
```

例:

```text
quiz-html-010/v0.1
quiz-css-020/v0.1
```

版管理の単位:

- 問題文
- 選択肢
- 正答
- 採点規則
- 解説
- 関連前提ノード
- 合格点

変更ルール:

- 正答や採点規則を変える場合は版を上げる。
- 表記修正のみでも、評価期間中は変更履歴を残す。
- 評価データには、受験時の`questionSetVersion`を含める。

## 20. 論理データ契約

本節は保存実装ではなく、KAI-11がKAI-10/KAI-12/KAI-15へ渡すべき論理契約である。

```ts
interface QuizAttempt {
  attemptId: string
  quizId: string
  nodeId: string
  attemptNumber: number
  questionSetVersion: string
  score: number
  maxScore: number
  passScore: number
  percentage: number
  passed: boolean
  incorrectQuestionIds: string[]
  startedAt: string
  completedAt: string
}

interface QuizState {
  nodeId: string
  quizId: string
  passed: boolean
  firstAttemptId: string
  latestAttemptId: string
  effectiveAttemptId: string
  questionSetVersion: string
}

interface QuizRouteEvidence {
  nodeId: string
  quizId: string
  passed: boolean
  effectiveAttemptId: string
  incorrectQuestionIds: string[]
}
```

`effectiveAttemptId`の扱い:

- 未合格状態では、最新の不合格受験を指す。
- 初めて合格した時点で、進行上の合格根拠となる受験を指す。
- 合格後の任意再受験が不合格でも、同一版では進行上の`effectiveAttemptId`を自動変更しない。
- 問題集合版変更後の再確認要否は、KAI-12/KAI-15および研究者判断で扱う。

[境界] `startedAt`、`completedAt`、保存先、保持期間、ログ出力形式、匿名研究IDはKAI-12で確定する。本書では、時刻情報が評価上必要になる可能性がある論理項目としてのみ示す。

## 21. 例外・欠損処理

| ケース | 扱い |
| --- | --- |
| 未受験 | 不合格とは扱わない。ルート生成へ失敗証拠として渡さない。 |
| 提出前の未回答 | 提出不可にするか、提出時に不正解扱いにするかはUI設計で決める。論理上は完了受験に未回答を含めるなら不正解とする。 |
| `nodeId`がMVP外 | 無効なテスト結果としてルート生成に渡さない。 |
| `questionId`が不明 | 主推薦は対象ノードのみとし、補助前提ノードは生成しない。 |
| `questionSetVersion`欠損 | 評価データとして不完全である。KAI-12で警告または保存拒否を検討する。 |
| 満点が3問でない | `score >= passScore`を使い、割合だけで合否を決めない。 |
| 採点不能 | 合格扱いにしない。必要なら再受験を促す。 |

## 22. KAI-12との境界

KAI-11で定義するもの:

- どの論理項目が必要か
- `passed`がどのように決まるか
- 不合格時にどの証拠をルート生成へ渡せるか
- 受験履歴を保持すべき理由

KAI-12で決めるもの:

- 保存先
- 保持期間
- 削除手順
- 同意取得
- 匿名研究IDと対応表
- 評価ログの出力形式
- `startedAt`、`completedAt`、`generatedAt`等の時刻付与責任
- 受験履歴をどこまで評価データとして扱うか

[注意] KAI-11の仕様案だけでは、評価データ収集の倫理・管理仕様は確定しない。

## 23. KAI-15との境界

KAI-11で定義するもの:

- ノードごとの到達目標
- 問題数の候補
- 許可する問題形式
- 採点と合格の規則
- 問題集合版と関連前提ノードの必要性

KAI-15で作成するもの:

- 実際の問題文
- 選択肢
- 正答
- 許容解
- 解説文
- 問題ごとの`relatedPrerequisiteNodeIds`
- 問題集合版の初期値
- 教材本文との整合確認

[注意] KAI-11で3問構成を採用候補にしても、KAI-15で各ノード3問の良問が作成できることは保証されない。作問困難なノードでは、問題数または到達目標を再検討する。

## 25. 具体例

### 25.1 3問中2問正答

```json
{
  "quizId": "quiz-html-010",
  "nodeId": "html-010",
  "score": 2,
  "maxScore": 3,
  "passScore": 2,
  "passed": true,
  "incorrectQuestionIds": ["html-010-q2"]
}
```

扱い:

- 合格として進行可能にする。
- `QUIZ_FAILED`の復習理由は生成しない。
- 誤答IDは評価分析または任意復習表示に使えるが、主ルートの不合格理由にはしない。

### 25.2 3問中1問正答

```json
{
  "quizId": "quiz-html-021",
  "nodeId": "html-021",
  "score": 1,
  "maxScore": 3,
  "passScore": 2,
  "passed": false,
  "incorrectQuestionIds": ["html-021-q1", "html-021-q3"]
}
```

扱い:

- 不合格として`html-021`を主復習候補にする。
- 誤答問題が`html-020`を前提候補に持つ場合、`html-020`を補助復習候補にする。
- `route-generation.md`の優先順位では、他の証拠と競合した場合に`QUIZ_FAILED`として扱う。

### 25.3 不合格後に合格

1回目が1/3で不合格、2回目が2/3で合格の場合:

- 1回目の`QuizAttempt`は削除しない。
- 2回目の受験により`QuizState.passed=true`にする。
- ルート生成上の確認テスト不合格理由は、対応テスト合格により解消できる。
- 評価分析では「再受験で合格」として扱える。

### 25.4 確認テスト不合格と実践課題エラーが同時にある場合

`html-022`の確認テストに不合格で、実践課題でも属性エラーがある場合:

- `QUIZ_FAILED`と`PRACTICE_ERROR`は別原因として保持する。
- 確認テスト合格で解消できるのは`QUIZ_FAILED`である。
- 実践課題エラーは、該当する実践課題の解決またはOQ-007で定義される規則に従う。

### 25.5 合格後に任意再受験して不合格

`css-011`に一度合格した後、任意再受験で1/3になった場合:

- 同一問題集合版では進行状態の`passed=true`を維持する。
- 最新受験は不合格として履歴に残す。
- 研究評価では、合格済み状態と最新不合格を混同しない。
- 予備試行でこのケースが多い場合、丸暗記、問題難度、フィードバック設計を再検討する。

## 26. 反証・リスク

| リスク | 発生条件 | 影響 | 検出方法 | 緩和策 | 仕様変更条件 |
| --- | --- | --- | --- | --- | --- |
| 3問では測定信頼性が低い | 到達目標が広い、偶然正答が多い | 合格判定の妥当性が弱い | 予備試行の正答率、誤答分布、実践課題エラー | 評価上は補助指標に限定 | 4問以上または必須問題導入を検討 |
| 2/3合格が甘すぎる | 重要概念を1問落としても合格する | 未理解のまま進む | 合格後の実践エラー、再質問 | 問題設計で重要概念を分散 | 重要概念問題を必須化または3/3へ変更 |
| 2/3合格が厳しすぎる | 初学者が1問ミスで不安になる | 離脱、過剰復習 | 離脱率、再受験回数、主観負担 | 解説と再受験を整える | 1/3合格や問題難度調整を検討 |
| 丸暗記で通過する | 同じ問題を無制限再受験する | 理解確認になりにくい | 短時間再受験、同一誤答の推移 | 同一到達目標内の別問題をKAI-15で検討 | 問題プール化または再受験間隔導入 |
| ノード別問題の品質差 | 作問者のばらつき、教材との不整合 | ノード間で合格難度が異なる | 指導教員レビュー、予備試行 | 作問テンプレートとレビュー観点を用意 | 問題修正またはノード到達目標修正 |
| 誤答から前提ノードを過剰推薦する | 問題に多くの関連ノードを付ける | ルートが重くなる | 推薦件数、学習者の迷い | 主推薦は対象ノード、補助は少数に限定 | 補助候補数上限を導入 |
| 未受験を不合格扱いする | データ欠損と失敗を混同する | 不要な復習推薦 | ログ監査 | 未受験と不合格を別状態にする | データ契約を修正 |
| 版管理なしで評価する | 評価期間中に問題を修正する | 結果比較ができない | 受験データの版欠損 | `questionSetVersion`必須化 | 版欠損データ除外を検討 |
| 合格後の任意失敗を無視しすぎる | 合格後に理解が崩れる | 進行状態が楽観的になる | 最新失敗率、実践課題エラー | 進行状態と評価履歴を分ける | 一定条件で再確認フラグを導入 |
| フィードバックが学習改善につながらない | 解説が正答提示だけになる | 再受験しても理解が深まらない | 主観評価、同一誤答再発 | 誤概念に対応した解説をKAI-15で作成 | 解説テンプレートを変更 |

文献と現行文書だけでは、次の判断は確定しない。研究者本人または指導教員の確認が必要である。

| 判断 | 採用候補 | 判断理由 | 確認すべきこと |
| --- | --- | --- | --- |
| 問題数 | 1ノード3問 | 初学者負担と実装負荷を抑えつつ、1問よりは複数証拠を得る。 | 3問で到達目標を表せるか。 |
| 合格点 | 3問中2問以上 | 形成的評価として進行不能を避ける。 | 甘すぎる、厳しすぎるリスク。 |
| 再受験 | 不合格後は無制限 | 復習と再挑戦を重視する。 | 丸暗記や過剰試行の扱い。 |
| 合格後任意失敗 | 同一版では進行合格維持 | 状態混乱を避ける。 | 最新失敗を評価上どう読むか。 |
| 不合格時主推薦 | 対象ノード | 不合格は当該ノード未達の直接証拠である。 | 前提不足との切り分け。 |
| 補助復習 | 問題別前提ノード候補 | 誤答の理由を少しだけ細分化する。 | 推薦が過剰にならないか。 |
| 問題形式 | 4形式まで | 初学者負担と実装可能性のバランスを取る。 | UI負担と採点揺れ。 |
| 版管理 | 必須 | 評価期間の再現性に必要である。 | 版付与責任と変更手順。 |

## 30. 予備試行・指導教員確認案

### 30.1 予備試行案

本実験前に、2から3ノードだけを使った小規模予備試行を行うことを推奨する。

確認項目:

- 3問が長すぎないか。
- 2/3合格が甘すぎないか、厳しすぎないか。
- 問題文が初学者に理解できるか。
- 選択肢に一意の正答があるか。
- 不合格後の復習推薦が自然か。
- 解説を読んだ後に同じ誤答が減るか。
- 再受験回数が過剰にならないか。
- 問題形式の操作が学習内容以外の負担になっていないか。

### 30.2 指導教員確認に出す要点

指導教員へは、次の順序で確認する。

1. 確認テストを形成的評価として扱い、能力測定の主指標にしない方針。
2. MVP 12ノードそれぞれに3問を置く方針。
3. 3問中2問以上を合格とする方針。
4. 不合格後は再受験を許可し、履歴を残す方針。
5. 不合格時の主推薦を対象ノードとし、誤答別に補助前提ノード候補を持つ方針。
6. 保存、同意、評価ログはKAI-12で別途確定する境界。
7. 実際の問題文と教材整備はKAI-15で別途実施する境界。

```markdown
### D-018 OQ-006確認テスト規則の採用
```
