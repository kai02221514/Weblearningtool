## 5. 用語定義

| 用語 | 定義 |
| --- | --- |
| 確認テスト | 各学習ノードの理解確認を目的とする短い形成的評価。総合能力測定ではない。 |
| ノード別テスト | `quiz-{nodeId}`として、特定の学習ノードに対応する確認テスト。 |
| 問題集合 | あるノードに紐づく問題のまとまり。版情報を持つ。 |
| 受験 | 学習者が問題集合を提出し、採点結果が得られる1回の試行。 |
| 合格 | 当該ノードの確認テストについて、研究者レビュー済み仕様案の判定規則を満たした状態。 |
| 不合格 | 受験済みで判定規則を満たさない状態。未受験とは区別する。 |
| 未受験 | まだ確認テストの提出結果がない状態。不合格として扱わない。 |
| 有効受験 | ルート生成に渡す`passed`状態の根拠となる受験。 |
| 補助復習候補 | 誤答した問題が前提知識に関係する場合に、主推薦とは別に示せる関連ノード候補。 |

## 6. 確認テストの目的

[研究者採用済み] 確認テストは以下に限定して使用する。

1. 形成的評価
2. 学習進行判定
3. 復習判定
4. ルート生成入力
5. 推薦理由の証拠
6. 研究評価時の補助指標

[非目的] 確認テスト単独で、学習者の能力全体、長期保持、実践力、学習効果を証明しない。

## 10. 問題数

[研究者採用済み・予備試行後に再確認] 原則として各ノード3問とする。

注意:

- 3問は精密な能力測定ではない。
- 1問の影響が大きい。
- 偶然正解の影響を受ける。
- 予備試行で問題数の妥当性を確認する。
- 必要なら4問または5問へ変更する。

## 11. 問題形式

[研究者修正採用済み] MVPで正式に使用する問題形式は、次の2種類に限定する。

1. 単一選択
2. 短いコード補完

MVP対象外:

- 複数選択
- コード並べ替え
- 自由記述
- 長いコード入力
- 自動実行による採点

複数選択とコード並べ替えは、KAI-15で必要性が確認された場合の拡張候補とする。

短いコード補完の正規化候補:

- 前後空白の除去
- 連続空白の正規化
- 改行の正規化
- 大文字・小文字の正規化
- 複数の許容解

具体的な正規化仕様はKAI-15または実装Issueで確定する。

## 12. 採点規則

[研究者採用済み] 各問題は1点、各ノードの満点は3点とする。割合は表示用の補助情報とし、合格判定には`score >= passScore`を使用する。

| 形式 | 正答条件 | 部分点 |
| --- | --- | --- |
| 単一選択 | 選択肢IDが正答IDと一致する。 | なし |
| 短いコード補完 | KAI-15または実装Issueで定めた正規化後、許容解集合のいずれかと一致する。 | MVPではなし |

[注意] 部分点を入れると評価解釈とルート接続が複雑になるため、MVPでは導入しない。

## 13. 合格判定

[研究者採用済み・予備試行後に再確認] 各ノード3問中2問以上正解で合格とする。

```text
passed = score >= 2
maxScore = 3
passScore = 2
```

割合は表示用の補助情報とし、合格判定には`score >= passScore`を使用する。

注意:

- 2/3は能力保証の基準ではない。
- 合格は「次へ進める暫定状態」である。
- 実践課題で同概念のエラーが出た場合は、合格済みでも復習候補になり得る。
- 予備試行で甘すぎる、厳しすぎる場合は変更する。

## 14. 再受験規則

[研究者採用済み・予備試行後に再確認] 不合格後の再受験を許可し、再受験回数に制限を設けない。

条件:

- 各試行を記録する。
- `attemptNumber`を保持する。
- 初回結果を上書きしない。
- 再受験前に、誤答解説、復習教材への導線、復習ノード推薦のいずれかを確認できるようにする。
- 極端な連続受験を予備試行で確認する。
- 再受験回数を研究評価上の補助指標として扱えるようにする。

[注意] 受験履歴の保存先、保持期間、削除、同意、匿名化はKAI-12で確定する。

## 15. 合格後再受験の扱い

[研究者修正採用済み] MVPでは、合格後の任意再受験機能を提供しない。再受験できるのは不合格状態の学習者だけとする。

合格後に復習したい場合は、確認テスト再受験ではなく以下を利用する。

- 教材の再閲覧
- 実践課題の再実施
- 将来追加する任意練習問題

合格後再確認テストは、将来機能または別Issueとして扱う。

## 16. 不合格時のルート接続

[研究者採用済み] 不合格時の主復習候補は、受験した対象ノード自身とする。

```text
failed quiz-html-021 -> mainReviewNodeId = html-021
```

`docs/architecture/route-generation.md`では、確認テスト不合格は`QUIZ_FAILED`として、対象ノードを復習候補にできる。KAI-11では、その入力となる`passed=false`と`nodeId`を定義する。

接続規則:

- 未受験は不合格ではない。未受験は`quizResults`に存在しない、または`QuizState`が未作成の状態として扱う。
- 不合格は`passed=false`としてルート生成入力へ渡せる。
- 主推薦は対象ノード自身とする。
- 補助復習候補は、誤答した問題に紐づく前提ノードから生成する。
- 前提未完了ノードを飛ばす推薦をしない制約は、ルート生成側の前提補完規則と合わせて満たす。

## 17. 誤答と復習ノードの対応

[研究者採用済み] 各問題に、次を持たせる。

```ts
interface QuizQuestionSpec {
  questionId: string
  nodeId: string
  relatedPrerequisiteNodeIds: string[]
}
```

誤答時は次の順で復習候補を作る。

1. 対象ノードを主推薦
2. 誤答問題に紐づく前提ノードを補助推薦
3. 重複を除外
4. MVP外IDを除外
5. ルート生成側の前提関係規則を適用

AIによる誤答原因の自動推定はMVPでは行わない。

## 18. 解説・フィードバック規則

[研究者採用済み] 各問題について、最低限以下を表示する。

- 正誤
- 学習者の回答
- 正答
- 短い解説
- 復習すべき概念
- 復習教材への導線

不合格者は、解説または復習導線を確認した後に再受験できるようにする。

[注意] 解説文と復習教材への導線はKAI-15で作成し、問題集合版と同じ版管理対象にする。

## 19. 問題集合の版管理

[研究者採用済み] すべての問題集合に`questionSetVersion`を付与する。

```text
questionSetVersion = quiz-{nodeId}/v{major}.{minor}
```

版管理対象:

- 問題文
- 選択肢
- 正答
- 許容解
- 採点規則
- 解説
- 関連前提ノード
- 合格点

評価期間中の変更は履歴を残す。

## 20. 論理データ契約

[研究者採用済み] 最低限、以下の構造を研究者承認済み仕様案として扱う。ただし、保存場所、保持期間、削除、同意、匿名化はKAI-12で確定する。

```ts
interface QuizAttempt {
  attemptId: string
  quizId: string
  nodeId: string
  attemptNumber: number
  questionSetVersion: string
  score: number
  maxScore: number
  passScore: number
  percentage: number
  passed: boolean
  incorrectQuestionIds: string[]
  startedAt: string
  completedAt: string
}

interface QuizState {
  nodeId: string
  quizId: string
  passed: boolean
  firstAttemptId: string
  latestAttemptId: string
  effectiveAttemptId: string
  questionSetVersion: string
}

interface QuizRouteEvidence {
  nodeId: string
  quizId: string
  passed: boolean
  effectiveAttemptId: string
  incorrectQuestionIds: string[]
}
```

`effectiveAttemptId`の扱い:

- 未合格状態では、最新の不合格受験を指す。
- 初めて合格した時点で、進行上の合格根拠となる受験を指す。
- QA-09により、MVPでは合格後の任意再受験を提供しない。
- 同一`questionSetVersion`について、一度`passed=true`になった後は維持する。
- 問題集合版変更後の再確認要否はKAI-12、KAI-15、または別Decisionで扱う。

## 21. 例外・欠損処理

| ケース | 扱い |
| --- | --- |
| 未受験 | 不合格とは扱わない。ルート生成へ失敗証拠として渡さない。 |
| 提出前の未回答 | 提出不可にするか、提出時に不正解扱いにするかはUI設計で決める。論理上は完了受験に未回答を含めるなら不正解とする。 |

| 満点が3問でない | `score >= passScore`を使い、割合だけで合否を決めない。 |
| 採点不能 | 合格扱いにしない。必要なら再受験を促す。 |

## 25. 具体例

### 25.1 3問中2問正答

```json
{
  "quizId": "quiz-html-010",
  "nodeId": "html-010",
  "score": 2,
  "maxScore": 3,
  "passScore": 2,
  "passed": true,
  "incorrectQuestionIds": ["html-010-q2"]
}
```

扱い:

- 合格として進行可能にする。
- `QUIZ_FAILED`の復習理由は生成しない。
- 誤答IDは評価分析または任意復習表示に使えるが、主ルートの不合格理由にはしない。

### 25.2 3問中1問正答

```json
{
  "quizId": "quiz-html-021",
  "nodeId": "html-021",
  "score": 1,
  "maxScore": 3,
  "passScore": 2,
  "passed": false,
  "incorrectQuestionIds": ["html-021-q1", "html-021-q3"]
}
```

扱い:

- 不合格として`html-021`を主復習候補にする。
- 誤答問題が`html-020`を前提候補に持つ場合、`html-020`を補助復習候補にする。
- `route-generation.md`の優先順位では、他の証拠と競合した場合に`QUIZ_FAILED`として扱う。

### 25.3 不合格後に合格

1回目が1/3で不合格、2回目が2/3で合格の場合:

- 1回目の`QuizAttempt`は削除しない。
- 2回目の受験により`QuizState.passed=true`にする。
- ルート生成上の確認テスト不合格理由は、対応テスト合格により解消できる。
- 評価分析では「再受験で合格」として扱える。

### 25.4 確認テスト不合格と実践課題エラーが同時にある場合

`html-022`の確認テストに不合格で、実践課題でも属性エラーがある場合:

- `QUIZ_FAILED`と`PRACTICE_ERROR`は別原因として保持する。
- 確認テスト合格で解消できるのは`QUIZ_FAILED`である。
- 実践課題エラーは、該当する実践課題の解決またはOQ-007で定義される規則に従う。

### 25.5 MVP対象外: 合格後再確認テスト

MVPでは、合格後の任意再受験機能を提供しない。将来、合格後に再確認テストを提供する場合は、別Issueまたは別Decisionで扱う。現時点では、同一問題集合版で合格後に不合格へ戻るケースは正式仕様に含めない。

### 30.2 指導教員確認に出す要点

指導教員へは、次の順序で確認する。

1. 確認テストを形成的評価として扱い、能力測定の主指標にしない方針。
2. MVP 12ノードそれぞれに3問を置く方針。
3. 3問中2問以上を合格とする方針。
4. 不合格後は再受験を許可し、履歴を残す方針。
5. 合格後の任意再受験をMVPでは提供しない方針。
6. 不合格時の主推薦を対象ノードとし、誤答別に補助前提ノード候補を持つ方針。
7. 保存、同意、評価ログはKAI-12で別途確定する境界。
8. 実際の問題文と教材整備はKAI-15で別途実施する境界。
9. 予備試行の3ノードと3〜5名程度という案。

## 32. Decision Log追記案

次の案は、指導教員確認、予備試行、OQ-006解消判断が終わった後に`docs/research/09-decision-log.md`へ追記する候補である。本PRでは追記しない。

```markdown
### D-018 OQ-006確認テスト規則の採用

- 状態: [案]
- 日付: YYYY-MM-DD
- 関連OQ: OQ-006
- 関連Issue: KAI-11
- 決定者: 研究者本人、指導教員確認後に正式化
- 決定:
  - 確認テストは形成的評価、学習進行判定、復習判定、ルート生成入力、推薦理由の証拠、研究評価時の補助指標に限定する。
  - MVP 12ノードに`quiz-{nodeId}`形式の確認テストを置く。
  - 原則として各ノード3問、3問中2問以上で合格とする。
  - 合格判定は割合ではなく`score >= passScore`で行う。
  - MVPで正式に使用する問題形式は、単一選択と短いコード補完の2種類に限定する。
  - 不合格後の再受験を許可し、受験履歴を保持する。
  - MVPでは合格後の任意再受験機能を提供しない。
  - 不合格時の主推薦は対象ノード自身とし、誤答別の補助前提ノード候補を保持する。
  - 問題集合版`questionSetVersion`を必須にする。
- 理由:
  - 確認テストを形成的評価とルート生成入力に限定するため。
  - 初学者の負担とMVP作問負荷を抑えるため。
  - `route-spec/1.0`の`quizResults[].passed`へ接続するため。
- 対象外:
  - 保存、保存期間、削除、同意、匿名研究ID、評価ログ、時刻付与責任はOQ-009/KAI-12で扱う。
  - 実際の問題文、選択肢、正答、許容解、解説、教材整備、短いコード補完の正規化はKAI-15で扱う。
- リスク:
  - 3問では測定信頼性が限定される。
  - 2/3合格の妥当性は予備試行または指導教員確認が必要である。
  - 無制限再受験は丸暗記のリスクがあるため、再受験回数を補助指標として扱う。
- 再検討条件:
  - 予備試行で合格率、誤答分布、再受験回数、実践課題エラー、解説理解、復習推薦の納得感に明確な偏りが出た場合。
  - 指導教員確認により問題数、合格点、予備試行人数、問題形式の見直しが必要になった場合。
```
