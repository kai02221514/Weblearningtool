# Weblearningtool Task Executor

Weblearningtoolのissue-sizedタスクを、リポジトリの正本とGit・GitHub規約に従ってDraft PRまで一貫して実行する。研究仕様そのものはこのSkillに保持せず、実行時点のリポジトリから確認する。

## 1. 作業コンテキストを特定する

1. 現在の依頼、Linear Issue、GitHub Issue、PR、ブランチから対象作業を特定する。
2. ユーザーが既に提示した情報を再質問しない。
3. 対象を一意に特定できない場合はコードや文書を変更せず、不足情報を報告する。
4. 調査・レビューだけの依頼では、明示的な変更依頼がない限り変更、commit、push、PR作成を行わない。

## 2. 正本と現状を確認する

実装前にリポジトリルートの `AGENTS.md` を読み、そこに記載された最新の参照順序に従う。最低限、タスクに関係する次の情報源を確認する。

- 現在の依頼またはIssue
- `docs/research/09-decision-log.md`
- `docs/research/01-confirmed-decisions.md`
- `docs/research/02-open-questions.md`
- その他の関連する研究、アーキテクチャ、運用文書
- `docs/development/git-workflow.md`
- `docs/operations/codex-workflow.md`
- `docs/operations/ai-research-development-roadmap.md`
- `docs/research/10-handover.md`
- 現在のコード、型、テスト、`package.json`

研究仕様の固定値や規則をこのSkillから補わない。ノードID、診断、ルート生成、確認テストなどの有効な規則は毎回正本から確認する。コード、mock、固定値、既存UIは実装状態の証拠として扱い、研究仕様の正解とは扱わない。`docs/references/` と `docs/archive/` は非正本として扱う。

## 3. 未確定事項を検出して停止する

次のいずれかに該当する場合は、実装や仕様確定を開始しない。

- タスクの受入条件達成または変更内容の確定に、`docs/research/02-open-questions.md` の未解決事項に対する判断が必要であり、確定済み範囲と未確定範囲を安全に分離できない。
- Decision Logで提案中、未承認、または失効した内容を前提にする必要がある。
- 受入条件を満たすために研究者の新しい判断が必要である。
- 研究方法、評価条件、個人データ、同意、保存、データ管理について指導教員または所定の判断者の確認が必要である。
- コードから研究仕様を推測しなければ変更できない。
- 正本間、またはIssueと正本の矛盾を解消できない。

関連する未確定事項が存在するだけでは停止しない。保存、同意、評価ログなどの未確定範囲を明示的に対象外とし、確定済み範囲だけで受入条件を満たせる独立作業は、境界と対象外を記録して継続する。

停止時は次を報告する。

- 未確定事項
- 判断が必要な理由
- 明示的に記録された判断者。記録がなければ「未確認」
- 決定後に再開できる作業
- Decision Logまたは対応する正本への反映先候補

判断者や承認状態を推測しない。研究目的、研究方法、評価方針、研究上の主張に関わる場合は、必要に応じて最新の明示的な指導教員合意、履修計画書、正式提出資料との整合も確認する。

## 4. 作業範囲を確定する

変更前に次を整理し、相互に矛盾しないことを確認する。

- 目的
- 根拠となる依頼、Issue、Decision、正本文書
- 変更対象
- 対象外
- 制約
- 受入条件
- 検証方法
- 研究上の影響
- 文書更新の要否

目的、変更対象、対象外、受入条件が不足し、軽微な前提補完では安全に確定できない場合は変更を開始しない。研究仕様変更とアプリケーション実装は、Issueが明示的に両方を要求しない限り分離する。

## 5. Gitの開始ゲートを通す

`docs/development/git-workflow.md` を正として次を実行する。

1. リポジトリルート、現在のブランチ、remote、worktree、最近のcommitを確認する。
2. 同一目的のSkill、ブランチ、open PRがないか確認する。
3. 無関係な既存変更があれば触れず、安全に分離できない場合は停止する。
4. default branchをリモートから確認し、fetch後にfast-forwardで最新化する。
5. default branchから規約どおりの `<type>/<short-description>` 作業ブランチを作る。
6. default branch上でファイル変更、commit、pushを行わない。

同一ブランチの既存PRがある場合は重複作成せず、そのPRを継続するかユーザーへ状況を報告する。

## 6. 変更前に調査する

タスクに応じて次を確認する。

- 関連コード、型定義、テスト
- 関連研究文書と承認済みアーキテクチャ文書
- 運用文書と過去の関連PR
- `package.json` に現在存在するscripts

ライブラリや外部APIの仕様が必要な場合は、公式文書または承認済みの取得手段で現行仕様を確認する。記憶だけでAPI契約を確定しない。

## 7. スコープ内だけを変更する

- Issueまたは依頼で明示された範囲だけを変更する。
- 研究の中核外の大規模変更、UI全面改修、技術移行を追加しない。
- 未確定仕様を埋めるdefault値や固定値を追加しない。
- 型安全性、明示的なエラーハンドリング、再現性を維持する。
- 研究ロジックを純粋関数として分離できる場合はUIから分離する。
- 既存の正本と重複する仕様文書を新設しない。
- `src/` はIssueがアプリケーション挙動またはテストを明示的に対象とする場合だけ変更する。

## 8. 定義済みコマンドで検証する

`package.json` と `docs/operations/codex-workflow.md` を実行時に確認し、そこに定義された現在の統合検証入口を優先する。現時点の入口は `npm run verify` だが、固定のコマンド構成を前提にせず、定義が変更された場合は最新の入口に従う。

対象機能の限定テストがある場合は統合検証より先に実行する。個別のtypecheck、lint、test、buildコマンドは、統合検証に含まれない場合、失敗原因を切り分ける場合、またはIssueが個別結果を要求する場合に、`package.json` に存在することを確認して実行する。変更内容にかかわらず `git diff --check` を実行する。UI変更では可能な範囲でブラウザ手動確認を行う。

文書またはSkillのみの変更でアプリケーション検証を省略する場合は、変更範囲に基づく理由を記録する。実行していない検証、失敗した検証、部分確認を成功として報告しない。

## 9. 文書更新の要否を判定する

変更が実装状態、確定済み仕様、未確定事項、評価計画、引継ぎ状態、運用方法に影響するか確認する。現在の依頼で新しい研究判断が明示された場合は、Decision Logまたは対応する正本へ反映してから完了するか、変更分離が必要なら反映案を提示する。

単なる実装で新しい研究判断がない場合は、不要なDecisionを追加しない。

## 10. 差分を確認してコミットする

1. `git status` とdiffを確認し、変更対象と無関係な差分を除外する。
2. stage対象を明示し、staged diffを確認する。
3. レビューとrevertが可能な論理単位でcommitする。
4. 大規模変更を単一commitへ押し込まない。
5. `<type>(<scope>): <日本語の変更要約>` 形式など、現行規約に従う。

## 11. pushしてDraft PRを作る

1. 作業ブランチをpushする。
2. 同一head branchの既存PRがないことを再確認する。
3. 必ず `gh pr create --draft` を使用する。
4. 作成後にPRの番号、URL、Draft状態、base/head、本文を確認する。

PR本文には最低限、次を記載する。

- 目的・背景
- 関連Issue・Linear・研究文書
- 変更内容
- 変更対象
- 対象外
- 検証方法と結果
- 未検証事項
- 研究上の影響
- リスク・注意点
- レビュー時の確認事項
- ロールバック方法

失敗または未実施の検証を明記し、研究仕様への影響がない場合と未検証事項がない場合もそれぞれ「なし」と記載する。PRはマージしない。

## 12. 最終報告を行う

完了後に次を簡潔かつ具体的に報告する。

- ブランチ名
- PR番号とURL
- commit一覧
- 変更ファイル一覧
- 実行したコマンドと各結果
- 未実施または失敗した検証
- 対象外として残した事項
- 発見した未確定事項
- Decision Logなど研究文書の更新要否
- ChatGPTによる監査で確認すべき点

PR作成まで完了できなかった場合は、完了した範囲、失敗したコマンド、原因、ユーザーが再開できる次の手順を報告する。
